KEY CONCEPTS EXPLAINED (One Sentence Each)
================================================================================

HAVERSINE FORMULA:
  Mathematical formula that calculates the great-circle distance between two
  points on Earth using their latitude/longitude coordinates.

GEOFENCING:
  Imaginary boundary (100m radius) around each work location; staff must be
  within this boundary to be verified.

SERVER-SIDE TIMESTAMP:
  Time is generated on the server (not the user's phone), making it impossible
  for users to manipulate by changing their device time.

VERIFICATION:
  System automatically checks: Is staff member at correct location AND within
  work hours? If yes: ✓ VERIFIED. If no: ✗ INVALID.

ANOMALY DETECTION:
  System checks if staff member could have physically traveled between two
  locations in the time shown; if speed exceeds ~100 km/h, it's flagged.

IMMUTABILITY:
  Once a record is saved to the database, it cannot be deleted or changed;
  database triggers prevent modifications.

AUDIT TRAIL:
  Complete log of all submissions, changes, and reviews; ensures accountability
  and provides evidence in case of disputes.

DEFENDING YOUR THESIS - PREPARED ANSWERS
================================================================================

Q: "Why don't you have real GPS satellites?"
A: "The system architecture is location-agnostic. I'm demonstrating the
   verification concept using simulated coordinates. The Haversine formula
   and verification logic work identically with real GPS, WiFi triangulation,
   or simulated data. This approach is standard in academic research."

Q: "Does this really prove the system works?"
A: "Absolutely. I'm proving four key mechanisms work correctly:
   1. Distance calculation (Haversine formula)
   2. Time validation (server-side timestamp)
   3. Anomaly detection (velocity check)
   4. Data integrity (database immutability)
   The source of location coordinates doesn't matter."

Q: "How would you deploy this with real GPS?"
A: "The code is ready. We'd simply:
   1. Switch from simulated to real HTML5 Geolocation API
   2. Deploy on HTTPS (required for browser GPS)
   3. Update database connection to server
   The verification logic doesn't change at all."

Q: "What prevents staff from cheating?"
A: "Three layers:
   1. Location: Haversine formula calculates exact distance (±0.01m precision)
   2. Time: Server timestamp (staff can't manipulate device time)
   3. Behavior: Velocity check (flags impossible travel patterns)
   Plus: Database triggers prevent tampering with records."
